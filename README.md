# jq-pm-copilot

企业产品经理第一阶段 Skill 工作台。它把产品发现、调研、PRD、原型、评审、汇报和企业 UI 规范串成一条可复用的交付链路。

## 第一阶段包含

| Skill | 适用场景 | 主要产出 |
| --- | --- | --- |
| jq-pm-discovery | 想法模糊、需求口径不一致、需要立项前澄清 | 产品发现简报、决策与待确认项 |
| jq-pm-research | 竞品、政策、用户、技术或行业调研 | 可信调研报告、产品启发与行动建议 |
| jq-pm-prd | 已有结论需要落成正式需求 | 可评审、可开发、可验收的 PRD |
| jq-pm-prototype | 需要把流程和页面做成可点击原型 | 原型文件、页面状态清单、PRD 差异记录 |
| jq-pm-review | 评审前压力测试 PRD 和原型 | 分级问题清单、章节级修改建议、评审结论 |
| jq-pm-briefing | 向领导、研发、测试、销售或群聊同步 | 可直接发送的汇报稿与短口径 |
| jq-ui-spec | 生成或审查企业产品界面 | 设计令牌、组件规范、页面模式和检查结果 |

## 推荐使用顺序

需求较模糊时，从 jq-pm-discovery 开始；需要外部证据时接 jq-pm-research；结论稳定后用 jq-pm-prd；随后用 jq-pm-prototype 和 jq-ui-spec 制作原型；评审前用 jq-pm-review；阶段同步用 jq-pm-briefing。

每个 Skill 都支持自动识别，也可以显式输入 $jq-pm-discovery、$jq-pm-prd 等名称调用。

## 安装

将整个仓库作为 Codex 插件安装，可获得统一入口。若团队环境暂未启用插件安装，也可以把 skills 目录下需要的 Skill 复制到个人 Codex skills 目录中试用。

安装后建议用下面三个真实任务做验收：

1. 用一个模糊产品想法生成产品发现简报。
2. 用一份现有 PRD 生成可点击原型并检查 UI 一致性。
3. 同时输入 PRD 和原型，生成正式评审修改清单。

## 设计原则

- 先读取项目既有材料，再形成结论。
- 明确区分事实、判断、建议和待确认项。
- 输出可直接评审、修改、发送或继续加工的成品。
- 遇到关键口径缺失时持续追问，避免代替业务做重大决定。
- 企业 UI 默认采用 PC 端后台产品基线，设计决策详见 jq-ui-spec。
