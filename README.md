# jq-pm-copilot

面向产品经理的 AI 协作工作台。当前包含 8 个互相衔接的 Skill，以第一性原理、产品拷问和产品上下文为底层能力，覆盖产品发现、专题研究、PRD 编写、交互原型、方案评审、沟通汇报和企业 UI 规范。

这套工作台适合持续推进真实项目。它会先读取项目材料、识别当前阶段和信息缺口，再产出可以继续评审、设计、开发、测试或汇报的成果。产品经理可以显式指定某个 Skill，也可以直接描述任务，由 Codex、Claude Code 或 WorkBuddy 根据上下文自动选择。

## 复制 GitHub 地址，让 AI 自己安装

把下面整段内容复制给 Codex、Claude Code 或 WorkBuddy。AI 应当识别当前平台，完成安装、依赖检查和结果汇报。

```text
请安装并启用下面 GitHub 仓库中的完整 jq-pm-copilot：

https://github.com/SuperChason/jq-pm-copilot

要求：
1. 使用当前平台官方支持的 Skill 或插件安装方式。
2. 安装全部 8 个 Skill，保留 references、assets、scripts 等配套资源。
3. 不要只介绍或总结仓库，直接完成安装。
4. 检查 jq-pm-grilling 共享引擎、第一性原理协议、跨 Skill 引用和平台清单。
5. 安装后执行仓库提供的自检，并报告平台、安装版本、成功项和未通过项。
6. 如果平台要求人工确认导入，只让我完成最后一个必要的确认动作。
```

更新时复制：

```text
请把我已经安装的 jq-pm-copilot 更新到下面 GitHub 仓库的最新正式版本，
保留完整的 8 个 Skill，更新后重新执行依赖和跨 Skill 自检，并报告结果：

https://github.com/SuperChason/jq-pm-copilot
```

裸 GitHub 地址容易被理解成“阅读这个仓库”。保留“安装完整套件并自检”这句指令，执行更稳定。

## 它解决哪些常见问题

- 想法只有一句话，目标用户、场景、价值和边界都没有说清。
- 方案主要来自竞品模仿、旧系统惯例或领导指定功能，目标与方案之间缺少可验证的因果链。
- 讨论形成了很多结论，决定、假设和术语没有持续沉淀，后续阶段反复从头确认。
- 调研资料很多，来源质量参差，最后很难转化成产品判断。
- PRD 看起来完整，研发、测试和业务仍然各有一套理解。
- 原型只有静态页面，缺少状态、异常、权限和关键交互。
- 评审会临近，产品经理需要提前发现阻塞开发和验收的问题。
- 同一件事要分别向领导、研发、测试、销售和项目群说明。
- 企业后台页面风格不统一，组件、布局和交互缺少共同基线。

       ↓
    jq-pm-review     在正式评审前发现阻塞与风险
       ↕
    jq-pm-grilling   逐题关闭会改变评审结论的问题
       ↓
    开发与验收

## jq-pm-grilling：产品拷问与上下文沉淀引擎

### 定位与核心价值

jq-pm-grilling 是整套工作台的底层推导与拷问引擎。它沿着产品决策树逐个处理高影响未知项，每轮只问一个问题，同时提供推荐答案和取舍。讨论形成决定时，它可以同步维护术语、决定、假设、第一性推导卡和长期决策记录，让结论直接流入发现、PRD、原型和评审。

### 第一性原理如何融入

这套方法不会机械套在每个任务上。出现以下情况时才启动：用户明确要求第一性原理；方案从功能或页面开始；主要理由是“竞品都有”“行业一直这样”或“旧系统就是这样”；一个决定会显著改变方向、首期范围、成本、风险或验收；目标与方案之间缺少可解释的因果链。

快速检查由当前 Skill 完成：

1. 写出要改变的用户或业务结果，移除预设功能。
2. 各列一条最关键的已验证事实、硬约束和待验证假设。
3. 检查“用户动作—产品机制—系统反馈—目标结果”能否形成最小因果链。

发现高影响断点后，jq-pm-grilling 进入深度推导：

1. 将输入拆成目标结果、已验证事实、硬约束、可协商约束、工作假设和惯例。
2. 检查约束由谁确认、何时成立、改变代价是什么，移除缺少依据的历史包袱。
3. 重建最小因果链，再比较零功能或人工流程、最小产品机制和完整方案。
4. 按目标贡献、证据强度、风险、成本和可逆性选择当前方向。
5. 定义最小验证、通过标准、最晚确认点和推翻当前决定的条件。
6. 把高影响决定沉淀为推导卡，并随产品上下文交给下游 Skill。

推导卡记录目标结果、事实、硬约束、已移除惯例、最小因果链、当前决定、替代方案、最小验证和推翻条件。第一性原理继续尊重已核验的用户证据、法律安全边界、领域规律和组织责任，停止条件是决定已经可以验证和交付。

它整合了原有 grilling、grill-me、grill-with-docs 和 domain-modeling 的核心能力：

- grilling 的逐题、持续、决策树式压力测试。
- grill-me 的轻量拷问入口。
- grill-with-docs 的边讨论边沉淀。
- domain-modeling 的统一术语、边界场景和决策记录。

### 两种使用模式

轻量拷问适合一次性讨论：

- 不主动创建项目文件。
- 在回复中维护精简决策账本。
- 达成共享理解后输出结论、假设和未决问题。
- 对应原有 grill-me 的使用体验。

带文档拷问适合持续项目：

- 在决定形成时即时更新项目文档。
- 维护规范术语、工作决定和工作假设。
- 对难逆转、缺少背景会令人意外且经过真实取舍的决定，按需创建 ADR。
- 对应原有 grill-with-docs 与 domain-modeling 的组合能力。

### 典型任务

- 在开始建设前彻底压力测试一个产品想法或方案。
- 对路线规划中的多个方向逐个做取舍。
- 解决范围、规则、权限、数据和验收之间的依赖关系。
- 在持续讨论中维护统一业务语言。
- 把用户回答即时沉淀成可供 PRD 和评审使用的项目事实。
- 对已有评审问题逐题关闭重大未知项。

### 建议提供的输入

- 当前产品想法、计划、设计或具体争议。
- 已有会议纪要、发现简报、PRD、原型或调研材料。
- 已经做出的决定和仍有分歧的内容。
- 硬性时间、资源、合规、数据和技术约束。
- 是否需要同步更新项目文档。

材料缺失时也可以开始。能够从项目文件、代码、数据或公开来源查明的事实，由 Skill 主动查找；价值取舍、风险承担和组织责任交给用户确认。

### 工作方式

1. 区分已确认事实、既有决定、工作假设和未知项。
2. 建立价值、用户、场景、结果、范围、规则、对象、权限、风险和验收的决策树。
3. 找出当前最上游、影响最大的未决节点，判断是否需要第一性原理推导。
4. 需要推导时拆分目标、事实、约束、假设和惯例，重建最小因果链。
5. 说明当前理解和该未知项的影响。
6. 给出推荐答案、备选方向和主要取舍。
7. 每轮只问一个问题，等待用户反馈。
8. 用反例、边界场景和推翻条件检验新决定。
9. 更新决策账本；带文档模式同步更新项目载体。
10. 剩余未知项不再改变方向、范围和验收，并且用户确认达到共享理解后停止。

### 产品上下文

整套 Skill 通过一组轻量产品上下文共享口径：

| 内容 | 默认载体 | 作用 |
| --- | --- | --- |
| 产品发现结论 | PRODUCT-BRIEF.md 或项目既有发现简报 | 保存用户、场景、问题、目标、范围和风险 |
| 规范术语 | CONTEXT.md、项目既有术语表，或 CONTEXT-MAP.md 指向的业务域术语表 | 统一对象、角色、状态和业务语言 |
| 工作决定 | docs/product/decision-log.md | 保存决定、理由、影响、状态和来源；高影响决定附推导卡 |
| 工作假设 | docs/product/assumptions.md | 保存错误代价、验证动作、责任人和最晚确认点 |
| 长期决定 | docs/adr/NNNN-slug.md | 保存真正难逆转且经过取舍的决定 |

这些文件全部按需创建。成熟项目继续沿用已有目录与命名，不强制迁移。根目录存在 `CONTEXT-MAP.md` 时，Skill 会先定位当前业务域及系统级影响，再分别维护对应术语和决定。

### 交付物

- 共享理解摘要。
- 已确认事实和决定。
- 工作假设及验证计划。
- 高影响决定的第一性推导卡。
- 规范术语及应避免的别名。
- 按影响排序的未决问题。
- 下一阶段交接。
- 带文档模式下的文件变更清单。

### 能力边界

- 不替产品经理做价值取舍和风险承担决定。
- 不把推荐答案当作用户确认结论。
- 不在一轮中堆叠多个问题。
- 不重复询问能够主动查明的事实。
- 不在达成共享理解前开始实施方案。
- 不为了留下文档创建大量空模板。

### 与其他 Skill 的协作

- jq-pm-discovery 调用发现拷问模式，结束后生成完整产品发现简报。
- jq-pm-review 调用评审拷问模式，结束后完成问题分级和评审结论。
- jq-pm-research 接收需要外部证据验证的假设。
- jq-pm-prd 读取决定、假设和规范术语，形成正式需求。
- jq-pm-prototype 使用相同术语、状态和业务决定。
- jq-pm-briefing 从当前有效上下文提取统一沟通口径。

### 示例提示词

- “用 jq-pm-grilling 逐题拷问这个 AI 审核产品想法，每次只问一个问题，并给出你的推荐答案。”
- “用第一性原理拆解‘竞品都在做 AI 自动审批，所以我们也要做’这个判断，从目标、事实和真实约束重新推导方案。”
- “使用 jq-pm-grilling 的带文档模式梳理这套审批规则，形成决定时同步更新术语、假设和决策记录。”
- “这份 PRD 还有几个重大争议，请逐题拷问会改变评审结论的问题，全部关闭后再回到正式评审。”

### 与原有 grill Skill 的对应关系

| 原有调用方式 | 当前建议 |
| --- | --- |
| grill-me | jq-pm-grilling 轻量拷问 |
| grilling | jq-pm-grilling，根据材料自动选择发现或评审上下文 |
| grill-with-docs | jq-pm-grilling 带文档拷问 |
| domain-modeling | jq-pm-grilling 带文档模式中的术语与决策维护 |

原有 Skill 可以继续单独使用。团队采用 jq-pm-copilot 时，优先使用 jq-pm-grilling，可以获得完整的产品阶段交接和统一触发规则。

上游 grill-me 当前位于 skills/productivity/grill-me，并保持显式调用：用户输入 $grill-me 时启动。jq-pm-grilling 作为本套件的底层引擎保留自动识别，两者触发策略互不干扰。当前上游基准为 mattpocock/skills 提交 697d4ce。

## jq-pm-discovery：产品发现与需求澄清

### 定位与核心价值

jq-pm-discovery 用于把模糊想法、零散反馈或互相冲突的需求，收敛成一份能够支持立项、调研或 PRD 编写的产品发现简报。它负责建立发现框架和综合结论；遇到会改变方向、范围或验收口径的重大未知项时，调用 jq-pm-grilling 逐题解决。

### 典型任务

- 老板提出一个方向，需要判断究竟要解决什么问题。
- 业务部门提交一批功能点，需要还原背后的业务目标。
- 多个角色对同一需求的理解不同，需要建立统一口径。
- 现有方案不断加功能，需要重新确定第一阶段边界。
- 立项前缺少成功指标、风险和关键假设。

### 建议提供的输入

- 原始需求、聊天记录、会议纪要或访谈记录。
- 已知的目标用户、业务流程和当前做法。
- 当前方案、历史版本或已有系统截图。
- 时间、预算、合规、技术和组织方面的约束。
- 已经做出的决定，以及仍然存在分歧的问题。

材料不完整也可以开始。Skill 会先区分已知事实、个人判断和待确认内容，再围绕高影响缺口提问。

### 工作方式

1. 读取现有材料，提取事实、角色、目标和已知限制。
2. 输入从指定功能或竞品做法开始时，先移除预设方案，按目标结果重新定义问题。
3. 复述当前理解，区分已验证事实、硬约束、可协商约束、惯例和隐含假设。
4. 找出会改变方向、范围或验收口径的重大未知项。
5. 调用 jq-pm-grilling 的发现拷问模式逐题解决。
6. 接收决策账本，将关键名词整理成统一术语。
7. 比较可选方向，说明各自收益、成本和风险。
8. 收敛第一阶段范围，记录明确排除项。
9. 输出发现简报和下一阶段交接。

### 交付物

- 背景与机会说明。
- 目标用户和关键场景。
- 核心问题及问题证据。
- 产品目标、成功指标和非目标。
- 第一阶段范围与排除项。
- 关键业务规则、约束和依赖。
- 核心术语表。
- 已确认决策、待确认问题和风险。
- 下一步调研或 PRD 建议。

### 能力边界

- 不会在关键信息缺失时替业务负责人做重大决定。
- 不会直接把所有想法包装成确定需求。
- 不替代用户访谈、业务验证和数据验证。
- 当外部证据影响方向选择时，会建议接续 jq-pm-research。

### 与其他 Skill 的协作

- 上游可以是会议纪要、用户反馈、管理层方向或存量方案。
- 重大未知项交给 jq-pm-grilling 逐题解决。
- 结论需要补证据时，交给 jq-pm-research。
- 结论稳定后，交给 jq-pm-prd。
- 需要同步发现阶段结果时，交给 jq-pm-briefing。

### 示例提示词

- “用 jq-pm-discovery 帮我梳理这个想法：做一个 AI 合同审核助手。先读取下面的会议纪要，再逐个追问会影响第一阶段范围的问题。”
- “业务提了 18 个功能点，我怀疑目标没有统一。请帮我还原用户、场景、痛点、价值和成功指标，最后给产品发现简报。”
- “我们对智能审批的第一阶段范围争议很大。请压力测试现有判断，明确必须做、可以延后和暂不进入的内容。”

## jq-pm-research：产品研究与可信证据

### 定位与核心价值

jq-pm-research 用于把一个产品问题转化成有边界、有来源、有判断的研究任务。它强调来源质量、事实与推断分离、证据可追溯，以及研究结论对产品决策的实际价值。

### 典型任务

- 竞品功能、定价、定位和产品路线研究。
- 行业趋势、政策变化或用户需求研究。
- 技术方案、平台能力和实现约束研究。
- 对一个产品判断进行证据验证。
- 为立项、路线规划或版本决策补充外部依据。

### 建议提供的输入

- 要支持的具体决策，例如“是否进入这个场景”。
- 研究对象、行业、地区和时间范围。
- 希望重点比较的维度。
- 已有资料、候选来源和内部判断。
- 交付时间、深度和受众。

### 工作方式

1. 先定义研究问题和它要支持的决策。
2. 拆出决定成立所需的底层事实和关键因果假设，再确定来源优先级和停止条件。
3. 优先查找官方文档、监管文件、产品页面和一手材料。
4. 平台支持并行 Agent 且任务可以独立拆分时，并行收集来源或对象资料；主 Agent 保留来源标准、冲突核验和最终综合。
5. 对关键事实进行交叉验证，记录发布日期和适用范围。
6. 区分来源事实、分析推断、产品建议和未知事项。
7. 比较差异、识别变化和异常。
8. 将研究发现转化为产品机会、风险和行动建议。
9. 保留来源链接和证据说明，方便复核。

### 交付物

- 研究目标、范围和方法。
- 核心结论摘要。
- 证据与来源表。
- 对象或方案对比。
- 趋势、变化和不确定性。
- 对产品规划的启发。
- 建议动作、验证方式和优先级。
- 研究限制和待补信息。

### 能力边界

- 不把搜索结果数量当作研究质量。
- 不使用无法核验的二手说法支撑关键结论。
- 不把推断写成已确认事实。
- 不自动把调研启发写进正式 PRD，产品经理确认后再进入需求定义。

### 与其他 Skill 的协作

- jq-pm-discovery 提供研究问题和信息缺口。
- 研究结论可以回填发现简报，也可以直接进入 jq-pm-prd。
- 重要发现需要对外同步时，交给 jq-pm-briefing。
- 需要研究竞品页面体验时，可以与 jq-ui-spec 配合。

### 示例提示词

- “用 jq-pm-research 调研国内外企业级 AI 审核产品，重点比较目标用户、核心流程、可解释性和人工复核机制。所有关键结论都要带来源。”
- “验证这个判断：企业用户更愿意先采用辅助审核，再逐步开放自动审批。请区分公开事实、合理推断和仍需内部验证的部分。”
- “研究近一年主流企业软件的 AI 助手交互变化，最后给出对我们下一版产品规划有用的 5 条启发。”

## jq-pm-prd：可执行的企业 PRD

### 定位与核心价值

jq-pm-prd 用于把已经收敛的产品结论转成可评审、可开发、可测试、可验收的正式需求。它使用通用 12 章骨架，以详细功能设计为正文核心，重点写清用户怎么使用、系统怎么反馈、业务如何判断、异常如何处理和什么结果可以验收。细节章节按实际需求选择，研发技术设计保留独立空间。

### 典型任务

- 从发现简报或会议结论生成正式 PRD。
- 补强只有功能描述、缺少业务规则的 PRD 初稿。
- 把复杂流程拆成研发和测试能够执行的需求。
- 为存量功能改造补齐影响范围和兼容要求。
- 在评审前统一业务、产品、研发和测试口径。

### 建议提供的输入

- 产品发现简报和研究结论。
- 当前业务流程、目标流程和相关角色。
- 业务规则、状态、数据口径和权限要求。
- 关联系统、已确认技术约束和历史兼容要求。
- 版本范围、优先级和发布时间。

### 工作方式

1. 检查需求目标、范围和关键决策是否稳定。
2. 让高影响需求可追溯到目标结果、已验证事实、真实约束和产品机制。
3. 建立角色、场景、术语和对象清单。
4. 描述现状流程、目标流程和变化点。
5. 按清晰用户任务或完整业务目标拆分功能。
6. 用一段话说明功能用途，再按需要补充界面图形、页面内容和功能说明。
7. 写清用户操作、系统反馈、业务规则、流程状态、权限、异常和验收结果。
8. 对多角色交接、条件分支和状态流转优先使用流程图或状态图。
9. 汇总产品影响、实施配套、场景验证和阻断待确认项。

### 交付物

- 文档信息和变更记录。
- 名词、背景、现状、关键需求、业务流程和功能清单。
- 按实际需求选择的授权场景控制点和主数据说明。
- 以功能概述、界面图形、数据域说明和功能说明为参考骨架的详细功能设计。
- 用户操作、系统反馈、业务规则、流程状态、权限、异常和可复现验收场景。
- 产品影响、实施体系配套和场景验证清单。

### 能力边界

- 不会把尚未确认的业务判断伪装成确定规则。
- 不用长篇背景掩盖功能、规则和验收细节。
- 不编写数据库字段、接口参数、数据表结构和研发实现方案。
- Skill 本身使用通用表达，不包含使用方公司名称、内部品牌和客户信息。
- 页面级体验需要进一步验证时，接续 jq-pm-prototype。

### 与其他 Skill 的协作

- 上游通常是 jq-pm-discovery 和 jq-pm-research。
- PRD 可以直接交给 jq-pm-prototype 生成原型。
- PRD 与原型完成后，用 jq-pm-review 做一致性和可执行性检查。
- 需要统一页面基线时，同时调用 jq-ui-spec。

### 示例提示词

- “根据这份产品发现简报生成正式 PRD，重点写清用户操作、业务规则、状态流转、异常处理和可测试的验收标准。复杂流程请画流程图。”
- “审阅这份 PRD 初稿，把所有依赖口头解释的地方补成明确规则。无法确认的内容集中放到待确认项，别自行假设。”
- “把这次规则配置改造写成增量 PRD，说明现状、目标、变化范围、历史数据兼容、权限影响和验收用例。”

## jq-pm-prototype：企业产品可点击原型

### 定位与核心价值

jq-pm-prototype 用于把 PRD 中的业务流程、页面状态和关键规则转成可点击原型。它关注完整任务链路和真实产品状态，产物适合方案演示、内部评审、可用性验证以及研发沟通。

### 典型任务

- 根据 PRD 生成企业后台产品原型。
- 将流程图和页面清单转成可操作界面。
- 补齐空态、加载、失败、无权限和边界状态。
- 对现有原型进行交互闭环检查。
- 快速制作可供领导或客户演示的高保真方案。

### 建议提供的输入

- PRD、业务流程和页面清单。
- 目标用户、核心任务和演示重点。
- jq-ui-spec 或其他明确的设计规范。
- 品牌、屏幕尺寸、技术栈和运行环境约束。
- 参考产品、现有页面和必须复用的组件。

### 工作方式

1. 从 PRD 提取角色、任务、页面、状态和关键规则，建立“任务—信息—状态—动作—反馈—页面”映射。
2. 建立信息架构和页面之间的跳转关系。
3. 确定最重要的端到端演示路径。
4. 依据 jq-ui-spec 选用布局、组件和视觉令牌。
5. 实现主流程、关键分支、反馈和状态变化。
6. 补齐空态、加载、失败、无权限和异常数据。
7. 检查原型与 PRD 的规则、字段和术语是否一致。
8. 运行原型并完成视觉、交互和基本可访问性检查。
9. 记录原型中发现的 PRD 缺口和设计决策。

### 交付物

- 可直接运行或打开的原型文件。
- 页面地图和核心任务路径。
- 页面、组件和状态清单。
- 关键交互说明。
- 与 PRD 的差异、假设和待确认事项。
- 运行方式和验收说明。

### 能力边界

- 原型用于验证产品和交互方案，不等同于生产代码。
- 不会为了视觉效果忽略真实业务规则和异常状态。
- 不会静默补写影响业务口径的规则。
- 对品牌或设计体系有强约束时，需要先提供对应规范。

### 与其他 Skill 的协作

- 上游通常是 jq-pm-prd。
- jq-ui-spec 提供设计令牌、组件、布局和交互基线。
- 原型和 PRD 一起交给 jq-pm-review。
- 演示前可以用 jq-pm-briefing 生成讲解稿。

### 示例提示词

- “用 jq-pm-prototype 根据这份 PRD 生成 PC 端企业后台可点击原型。必须覆盖新建、编辑、提交、失败、无权限和空数据状态。”
- “把这份流程图转成可演示原型，优先完成从任务列表到查看结果再到人工复核的完整链路。”
- “检查现有原型是否完整表达 PRD，列出缺失页面、无反馈交互、状态断点和需要产品确认的规则。”

## jq-pm-review：正式评审前的压力测试

### 定位与核心价值

jq-pm-review 用于在正式评审、开发排期或验收前，对 PRD、原型和相关材料进行压力测试。它会从业务闭环、逻辑一致性、角色权限、数据状态、异常处理、可实现性和可验收性等角度发现问题，并给出可以直接修改的建议。

### 典型任务

- PRD 正式评审前的完整性检查。
- PRD 与原型的一致性检查。
- 对复杂业务方案做反例和边界压力测试。
- 开发前检查需求是否具备实施条件。
- 测试前检查验收标准和异常场景是否充分。

### 建议提供的输入

- 当前准备评审的 PRD 和原型。
- 业务规则、领域术语和关键流程。
- 接口、数据、权限或技术约束。
- 本次评审目的、参会角色和必须达成的结论。
- 已知争议点和时间约束。

### 工作方式

1. 确认材料版本、评审目标和范围。
2. 建立目标、场景、需求、页面和验收的追溯关系。
3. 检查目标无因果关系、假设当事实、人为约束和过度方案，再检查流程闭环、规则冲突、状态遗漏和权限漏洞。
4. 对关键假设构造反例、极端情况和失败路径。
5. 检查 PRD 与原型的字段、术语、动作和状态是否一致。
6. 会改变评审结论的未知项交给 jq-pm-grilling 逐题关闭。
7. 按 P0 至 P3 对问题分级，说明影响和证据。
8. 给出章节级、页面级或字段级修改建议。
9. 输出通过、有条件通过或不通过的评审结论。

### 交付物

- 评审范围和总体结论。
- 分级问题清单。
- 每个问题的证据、影响和建议修改方式。
- PRD 与原型差异清单。
- 待决策事项和责任角色建议。
- 复审条件和评审结论。

### 能力边界

- 不会只给“建议优化”这类无法执行的意见。
- 不会把个人偏好升级成高优先级问题。
- 不直接替产品经理改写全部方案，除非用户明确要求修改。
- 缺少关键材料时，会先指出评审可信度受限。

### 与其他 Skill 的协作

- 输入通常来自 jq-pm-prd、jq-pm-prototype 和 jq-ui-spec。
- 会改变评审结论的重大未知项交给 jq-pm-grilling。
- 发现目标或范围根本不清时，可以退回 jq-pm-discovery。
- 评审结论需要向不同角色同步时，交给 jq-pm-briefing。

### 示例提示词

- “用 jq-pm-review 对这份 PRD 做正式评审前检查。按 P0 到 P3 分级，每个问题都写清位置、证据、影响和具体修改建议。”
- “同时检查 PRD 和原型，重点找字段不一致、状态缺失、权限漏洞、异常路径和无法验收的描述。”
- “站在业务负责人、研发负责人和测试负责人的角度压力测试这个方案，最后给出是否可以进入开发的明确结论。”

## jq-pm-briefing：面向不同受众的产品沟通

### 定位与核心价值

jq-pm-briefing 用于从项目材料中提取准确口径，并根据受众和沟通目的形成可以直接使用的汇报内容。它会调整信息顺序、篇幅和关注重点，让领导、研发、测试、销售、客户或项目群快速获得与自己相关的信息。

### 典型任务

- 向领导汇报阶段成果、风险和需要拍板的事项。
- 给研发和测试同步需求变化与影响范围。
- 在项目群发送简洁的进展和后续安排。
- 为演示、评审、周报或月报准备讲稿。
- 将同一份项目材料改写成多个受众版本。
- 在切换会话、任务或 Agent 前生成可直接接手的交接材料。

### 建议提供的输入

- PRD、汇报材料、会议纪要或项目进展记录。
- 目标受众及其最关心的问题。
- 沟通目的，例如知会、决策、争取资源或推动执行。
- 时间长度、字数、渠道和正式程度。
- 必须保留的事实、数据和风险。

### 工作方式

1. 先确认受众、目的、渠道和希望对方采取的动作。
2. 从项目材料中抽取已确认事实和最新口径。
3. 重大决定存在推导卡时，提取目标、事实、硬约束、选择、最小验证和推翻条件。
4. 选择对当前受众最重要的结果、变化、风险和决策。
5. 按结论优先的方式组织内容。
6. 删除无关细节，保留必要背景和证据。
7. 检查数字、时间、版本、责任人和行动项。
8. 根据需要提供正式版、口头版或极简版。

跨会话交接时，引用已有成果路径或链接，记录当前状态、验证结果、未决问题和下一步首个动作，并完成敏感信息脱敏与下一 Skill 推荐。平台支持临时文件时保存到操作系统临时目录。

### 交付物

- 可直接复制发送的短消息。
- 领导汇报要点或完整讲稿。
- 评审开场、演示串词和结论口径。
- 周报、月报和阶段总结。
- 风险说明、资源请求和决策请求。
- 针对不同受众的多版本表达。
- 跨会话交接文档。

### 能力边界

- 不会为了表达效果虚构成果、数据或确定性。
- 不会把项目材料中的过期口径当作当前事实。
- 不替代正式 PRD、决策记录或项目计划。
- 信息冲突时会标出差异，避免选择性省略关键风险。

### 与其他 Skill 的协作

- 可以读取其他 7 个 Skill 的任何阶段产物。
- 发现材料本身存在逻辑问题时，建议先回到对应 Skill 修正。
- 汇报中涉及重大判断但缺少证据时，可以接续 jq-pm-research。

### 示例提示词

- “用 jq-pm-briefing 从这些材料中整理一版领导汇报，控制在 5 分钟，突出已完成成果、当前风险和需要领导拍板的两件事。”
- “把这次需求变化整理成项目群通知，研发和测试能直接看懂影响范围、更新时间和各自下一步动作。”
- “同一份项目进展分别生成领导版、研发版和销售版，每版都保持事实一致，关注点按受众调整。”

## jq-ui-spec：企业后台 UI 生成与审查规范

### 定位与核心价值

jq-ui-spec 提供企业后台产品的视觉和交互基线，并支持两类任务：指导新页面生成，以及审查已有页面是否符合规范。它覆盖设计令牌、布局、导航、表格、表单、状态反馈、弹窗、空态和常见页面模式。

### 典型任务

- 为新的企业后台页面建立统一视觉和交互规则。
- 指导 jq-pm-prototype 生成符合企业产品习惯的原型。
- 审查现有 HTML、原型或截图中的 UI 问题。
- 建立列表、详情、配置、工作台等常见页面模式。
- 检查颜色、间距、字号、组件和操作层级的一致性。

### 建议提供的输入

- 页面目标、用户角色和核心任务。
- 页面清单、线框图、PRD 或现有原型。
- 品牌色、Logo、字体和必须遵循的既有规范。
- 屏幕尺寸、浏览器和可访问性要求。
- 需要重点检查的页面或组件。

### 工作方式

生成页面时：

1. 从用户任务、必要信息、动作、反馈和可访问性推导页面层级，避免惯性照搬旧系统或竞品。
2. 应用统一的颜色、字号、间距、圆角和阴影令牌。
3. 选择适合企业后台的导航、内容区和页面布局。
4. 使用规范组件表达查询、操作、数据和反馈。
5. 补齐加载、空态、错误、成功、禁用和无权限状态。
6. 检查密度、可读性、对齐、操作层级和响应反馈。
7. 使用随 Skill 提供的静态检查工具完成基础验证。

审查页面时：

1. 读取页面、代码、截图或原型。
2. 按设计令牌、布局、组件和交互规则逐项检查。
3. 区分阻塞问题、一致性问题和体验建议。
4. 指出具体位置、违反的规则和修正方式。
5. 修改后再次检查，确认问题关闭。

### 交付物

- 设计令牌和基础样式建议。
- 页面布局和信息层级建议。
- 组件选择、状态和交互说明。
- 常见页面模式的落地方案。
- UI 审查问题清单和修正建议。
- 静态检查结果。

### 能力边界

- 提供通用企业后台基线，不取代组织已有的正式设计系统。
- 有明确品牌规范时，以用户提供的规范为最高优先级。
- 不负责定义业务规则、字段含义和权限逻辑。
- 只检查视觉规则无法保证完整可用性，关键流程仍需真实用户验证。

### 与其他 Skill 的协作

- jq-pm-prd 提供业务、字段、权限和状态要求。
- jq-pm-prototype 使用本 Skill 的规范生成界面。
- jq-pm-review 结合本 Skill 检查原型质量和一致性。
- jq-pm-research 可以提供竞品体验和行业模式参考。

### 示例提示词

- “用 jq-ui-spec 为这个企业后台原型建立页面规范，给出颜色、字号、间距、表格、表单、按钮和反馈状态的具体要求。”
- “审查这组页面截图，按严重程度列出布局、层级、组件、颜色、间距和交互反馈问题，并给出可执行的修正方案。”
- “结合 jq-ui-spec 和 jq-pm-prototype 生成一个规则配置页面，重点保证复杂表单、条件组合、校验反馈和保存流程清晰。”

## 常用组合工作流

### 组合零：单独拷问一个产品计划

1. 用 jq-pm-grilling 读取已有材料并建立决策树。
2. 遇到竞品模仿、历史惯例、人为约束或因果断点时，启用第一性原理推导层。
3. 每轮只解决一个最高影响未知项。
4. 需要持续沉淀时切换到带文档模式。
5. 形成共享理解后，根据任务进入 jq-pm-discovery 或 jq-pm-review。

适合用户明确提出“grill me”“拷问这个方案”或希望在正式出稿前先把关键决定想透的任务。

### 组合一：从一句想法到可评审 PRD

1. 用 jq-pm-discovery 建立产品发现框架。
2. 用 jq-pm-grilling 逐题解决会改变方向和范围的未知项。
3. 将需要外部证据的假设交给 jq-pm-research。
4. 产品经理确认方向和边界。
5. 用 jq-pm-prd 读取决定、假设和术语，生成正式需求。
6. 用 jq-pm-review 检查是否具备评审条件。

适合新产品方向、新模块立项和需求范围争议较大的任务。

### 组合二：从 PRD 到可点击原型

1. 用 jq-pm-prd 检查页面、字段、状态、权限和异常要求。
2. 用 jq-ui-spec 确定布局、组件和视觉基线。
3. 用 jq-pm-prototype 实现完整任务链路。
4. 用 jq-pm-review 对照 PRD 检查一致性。
5. 用 jq-pm-briefing 准备评审讲稿或演示串词。

适合复杂企业后台、配置中心、工作台、审批和审核类产品。

### 组合三：竞品研究转产品规划

1. 用 jq-pm-discovery 明确研究要支持的产品决策。
2. 用 jq-pm-grilling 把产品判断拆成可验证假设。
3. 用 jq-pm-research 获取和验证证据。
4. 将启发、风险和未知事项交回产品经理判断。
5. 已确认的方向用 jq-pm-prd 落成需求。
6. 用 jq-pm-briefing 形成领导汇报或团队同步材料。

### 组合四：正式评审前一轮完整检查

1. 同时提供 PRD、原型、业务规则和必要技术约束。
2. 用 jq-pm-review 找出 P0 至 P3 问题。
3. 用 jq-pm-grilling 逐题关闭会改变评审结论的重大未知项。
4. 用 jq-ui-spec 补充界面一致性检查。
5. 回到 jq-pm-prd 或 jq-pm-prototype 修正。
6. 再次调用 jq-pm-review，直到满足复审条件。

## 两种调用方式

### 自动识别

直接描述你的真实任务，当前平台的 AI 助手会根据材料和目标选择合适的 Skill。例如：

> 我只有一段会议纪要，帮我把这个产品方向梳理清楚。

> 竞品都有这个功能，请用第一性原理判断我们是否真的需要，并重新推导最小方案。

> 每次只问我一个问题，彻底拷问这个产品方案。

> 这份 PRD 明天评审，帮我找出会卡住研发和测试的问题。

> 根据 PRD 做一个企业后台可点击原型，风格统一、状态完整。

当任务跨越多个阶段时，当前平台的 AI 助手会按交付链路组合调用，并说明当前使用了哪些 Skill。

### 显式调用

希望指定方法或稳定复现时，在任务中写出 Skill 名称：

> 使用 jq-pm-discovery 梳理下面的原始需求。

> 使用 jq-pm-grilling 的带文档模式逐题拷问，并同步维护决定、假设和术语。

> 同时使用 jq-pm-prototype 和 jq-ui-spec 完成原型。

> 使用 jq-pm-review 做正式评审前检查。

显式调用适合团队建立标准流程、编写任务模板和复用固定验收方式。

各平台也支持自己的显式语法：

- Codex：输入 `$jq-pm-discovery`，或从 Skill 列表中选择。
- Claude Code：输入 `/jq-pm-copilot:jq-pm-discovery`；其他 Skill 使用相同命名空间。
- WorkBuddy：在任务中直接写 Skill 名称，或从技能面板中选择对应 Skill。

## 怎样提供材料，效果更好

- 说明这次任务要支持什么决定或动作。
- 尽量区分已经验证的事实、不可突破的硬约束、可以协商的限制和自己的暂定假设。
- 提供当前有效版本，避免混入过期资料。
- 标明已经确认、仍有争议和暂时未知的内容。
- 说明受众、时间、范围和不可突破的约束。
- 有 PRD、原型、截图、会议纪要或数据时直接附上。
- 希望交付成文件时，说明文件格式、存放位置和命名要求。

材料不完整时可以直接开始。Skill 会优先读取已有内容，在真正影响结果的地方提出问题。

## 安装

### 推荐方式：把仓库地址交给 AI

优先复制 README 开头的安装提示词。AI 会根据当前平台选择下面的安装路径，并在完成后自检。

官方支持对象是完整套件。单独安装适合了解依赖关系的高级用户；产品发现、研究、PRD、原型、评审和汇报都会使用 jq-pm-grilling 的产品上下文协议。

### Codex

Codex 可以从 GitHub 仓库添加 Marketplace，再安装完整插件。将 GitHub 地址交给 Codex 后，它可以执行：

```bash
codex plugin marketplace add SuperChason/jq-pm-copilot --ref main
codex plugin add jq-pm-copilot@jq-pm-copilot
```

当前网络或代理导致 Codex 无法直接拉取 Marketplace 时，让 AI 自动采用本地快照兜底：

```bash
git clone --branch v0.4.2 --depth 1 https://github.com/SuperChason/jq-pm-copilot.git
codex plugin marketplace add ./jq-pm-copilot
codex plugin add jq-pm-copilot@jq-pm-copilot
```

安装后开始一个新任务，让新 Skill 进入上下文。更新时由 Codex 刷新 Marketplace，再重新安装当前正式版本。

### Claude Code

Claude Code 可以把同一仓库添加为插件 Marketplace。在 Claude Code 中执行：

```text
/plugin marketplace add SuperChason/jq-pm-copilot
/plugin install jq-pm-copilot@jq-pm-copilot
/reload-plugins
```

安装后的显式调用格式为 `/jq-pm-copilot:Skill名称`，例如：

```text
/jq-pm-copilot:jq-pm-grilling
/jq-pm-copilot:jq-pm-prd
/jq-pm-copilot:jq-pm-review
```

### WorkBuddy

先把安装提示词和 GitHub 地址发给 WorkBuddy。具备 GitHub 下载和本地操作权限时，让 WorkBuddy 直接下载最新 Release 并导入。

平台要求手动上传时：

1. 打开 GitHub 最新 Release。
2. 下载 `jq-pm-copilot-workbuddy-v版本号.zip`。
3. 解压后得到 8 个独立 Skill ZIP。
4. 在 WorkBuddy 的“添加技能 → 上传技能”中导入全部 8 个 ZIP。
5. 确认 8 个 Skill 都已启用，再执行安装验收任务。

WorkBuddy 包由发布脚本从同一份 `skills/` 源码自动生成，并移除仅供 Codex 使用的 `agents/openai.yaml`。

### 手动安装单个 Skill

如果只需要一项能力，可以复制 `skills/` 下对应目录，确保 `references`、`assets` 和 `scripts` 一起保留。

以下 Skill 依赖 jq-pm-grilling 的产品上下文协议：

- jq-pm-discovery
- jq-pm-research
- jq-pm-prd
- jq-pm-prototype
- jq-pm-review
- jq-pm-briefing

单独安装上述任一 Skill 时，同时安装 jq-pm-grilling。jq-pm-prototype 还需要 jq-ui-spec 才能获得完整的企业 UI 基线和静态检查能力。

### 安装自检

仓库根目录提供统一检查：

```bash
python3 scripts/validate-suite.py
```

检查内容包括：

- Codex 与 Claude Code 版本和 Marketplace 标签一致。
- 8 个 Skill 目录、元数据和配套资源完整。
- 跨 Skill 引用指向真实文件。
- 所有 Skill 已接入跨平台能力降级协议。
- 所有 Skill 已接入第一性原理产品推导协议。
- 跨平台验收题覆盖全部 8 个 Skill。
- 公开仓库中不含已配置的公司品牌词。

构建正式发布包：

```bash
python3 scripts/build-release.py
```

会生成：

- `dist/jq-pm-copilot-v版本号.zip`
- `dist/jq-pm-copilot-workbuddy-v版本号.zip`
- `dist/SHA256SUMS.txt`

### 安装后的验收任务

建议用七个真实任务确认安装和识别正常：

1. 输入一个产品方案，要求每次只问一个问题并维护决策账本。
2. 输入一句“竞品都在做，所以我们也要做”，要求使用第一性原理重新推导。
3. 输入一个模糊产品想法，要求生成产品发现简报。
4. 输入一份现有 PRD，要求生成可点击原型并检查 UI 一致性。
5. 同时输入 PRD 和原型，要求生成分级评审问题清单。
6. 在存在 `CONTEXT-MAP.md` 的项目中，要求区分系统级和业务域术语与决定。
7. 要求切换会话并生成引用已有成果、完成脱敏的交接材料。

验收时重点检查：

- 是否自动选择了正确的 Skill。
- “grill”类任务是否进入 jq-pm-grilling，并且每轮只有一个问题。
- “第一性原理”类任务是否区分目标、事实、约束、假设和惯例，并给出最小验证与推翻条件。
- 是否先读取材料再形成结论。
- 是否区分事实、判断、建议和待确认项。
- 交付物是否可以继续评审、修改、演示或发送。
- 关键缺口是否被明确提出。

仓库中的 `evals/cross-platform-cases.json` 保存了统一验收题。Codex、Claude Code 和 WorkBuddy 使用同一组输入，比较 Skill 选择、关键行为和能力降级结果。

### 平台能力不足时

8 个 Skill 共用跨平台能力降级协议：

- 没有文件写入能力时，返回可直接保存的完整成稿和建议文件名。
- 没有联网能力时，标明待核验结论，不编造最新事实和来源。
- 没有浏览器能力时，执行静态走查并提供人工验收清单。
- 不能运行脚本时，按同一规则人工检查并说明未执行项。
- 无法读取相邻 Skill 时，先完成当前阶段可交付内容，并建议补齐完整套件。

AI 在收尾时应区分实际执行动作、降级方式和仍需人工完成的验证。

### 权限与数据安全

- 第三方 Skill 可能读取本地文件、运行脚本或访问网络，安装前先确认仓库来源。
- 处理内部材料前检查当前平台的数据边界、工作空间策略和联网权限。
- 对外分享 PRD、原型、调研和项目材料前，移除客户名称、账号、接口地址和未公开业务数据。
- 权限不足时只开放完成当前任务所需的最小范围。

## 使用原则

- 先读取项目既有材料，再形成结论。
- 高影响决定先拆目标结果、已验证事实、真实约束、工作假设和历史惯例，再重建最小因果链。
- 明确区分事实、判断、建议和待确认项。
- 输出可以继续流转的成品，减少只有概念没有动作的内容。
- 遇到关键口径缺失时持续追问，保留产品经理的决策权。
- 用产品上下文协议贯通术语、决定、假设和阶段交接。
- 每个阶段都设置清晰的完成条件和下一步。
- 企业 UI 默认采用 PC 端后台产品基线，具体规则以 jq-ui-spec 为准。
