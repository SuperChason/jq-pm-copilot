# 第一性原理产品推导协议

把高影响产品判断拆到可验证事实、真实约束和最小因果链，再重新推导方案。将第一性原理作为决策工具使用，避免把它写成口号或无限追问。

## 触发条件

出现以下任一情况时执行本协议：

- 用户明确要求使用第一性原理、从底层推导或重新思考。
- 方案从功能或页面开始，目标用户、期望结果或问题证据仍不清楚。
- 理由主要是“竞品都有”“行业一直这样”“旧系统就是这样”或“领导要求做这个功能”。
- 一个决定会显著影响方向、首期范围、成本、合规、安全、数据责任或验收口径。
- 目标与方案之间缺少可解释的因果链，或成功指标只能证明产出数量，不能证明问题改善。
- 团队把预算、流程、技术、组织习惯或历史实现全部当成不可改变的硬约束。

普通格式整理、已确认规则的机械落稿和低风险界面细节默认不执行完整推导。先做快速检查；发现高影响未知项时，再进入 jq-pm-grilling 深度推导。

## 基础分类

先把输入拆成以下六类，不允许混写：

| 类别 | 判断问题 | 处理方式 |
| --- | --- | --- |
| 目标结果 | 哪个用户或业务结果必须发生可观察变化 | 写成结果，不预设功能 |
| 已验证事实 | 有什么证据能够复核，适用范围和日期是什么 | 记录来源与边界 |
| 硬约束 | 哪些法律、物理、合同、安全或已确认资源条件不能突破 | 保留并说明依据 |
| 可协商约束 | 哪些时间、预算、流程或系统限制可以通过取舍改变 | 标明确认权和改变代价 |
| 工作假设 | 哪些判断目前缺少证据，错误时会造成什么影响 | 进入假设清单和验证计划 |
| 惯例或类比 | 哪些做法仅来自竞品、旧系统、行业习惯或历史方案 | 暂时移除，重新验证必要性 |

将用户反馈、竞品做法和行业经验视为重要证据，继续核验其适用性。第一性推导不能代替用户研究、市场研究、法律判断和领域专家确认。

## 推导流程

1. **定义目标结果**：用“谁在什么场景下，要把什么结果从当前状态改变到目标状态”描述问题，移除功能名和实现方式。
2. **建立事实底座**：列出已验证事实及来源；把缺少证据的内容降级为假设；把约束分成硬约束、可协商约束和人为惯例。
3. **挑战隐含前提**：逐项追问该约束由谁确认、何时成立、取消后会发生什么。只挑战缺少依据的限制，不重复质疑已核验的法律、安全和物理边界。
4. **重建最小因果链**：说明用户采取什么动作，系统或组织提供什么机制，产生什么反馈，最终怎样影响目标结果。因果链中任何一环缺少依据时，明确标为假设。
5. **生成不同形态的方案**：至少比较零功能或人工流程、最小产品机制和完整方案。确保选项在机制上有差异，避免只比较界面样式。
6. **选择与验证**：按目标贡献、证据强度、风险、成本和可逆性选择当前方案；定义最小验证、通过标准、最晚确认点和推翻当前决定的条件。

## 快速检查与深度推导

### 快速检查

在 discovery、research、PRD、prototype、review、briefing 和 UI 任务中执行：

1. 写出目标结果。
2. 各列一条最关键事实、硬约束和待验证假设。
3. 检查当前方案是否能通过清晰因果链影响目标；不能时暂停扩写并转入深度推导。

### 深度推导

由 jq-pm-grilling 逐题执行完整协议。每轮只解决一个最高影响前提，给出推荐答案、替代方向和取舍；用户确认后更新决策账本和产品上下文。

## 推导卡

对会改变方向、首期范围、重大风险或验收口径的决定，输出并维护一张精简推导卡：

- 目标结果：要改变的用户或业务结果。
- 已验证事实：来源、日期和适用范围。
- 硬约束：依据及不可突破的原因。
- 已移除惯例：不再默认成立的历史做法或类比。
- 最小因果链：动作、机制、反馈与结果。
- 当前决定：从事实和约束推导出的选择。
- 替代方案：未选择的机制及主要取舍。
- 最小验证：动作、指标、通过标准、责任人和最晚确认点。
- 推翻条件：出现什么证据时重新打开决定。

将推导卡并入 decision-log 或项目已有决定记录，不额外创建同类文件。低风险、可逆决定保留精简记录即可。

## 各 Skill 的落点

| Skill | 第一性原理检查重点 |
| --- | --- |
| jq-pm-grilling | 逐题拆解前提、重建因果链、形成推导卡 |
| jq-pm-discovery | 从目标结果定义问题，移除预设功能，重构首期最小闭环 |
| jq-pm-research | 为底层事实和关键因果假设找证据，不把竞品清单当结论 |
| jq-pm-prd | 让关键需求可追溯到目标、事实、约束和产品机制 |
| jq-pm-prototype | 让每个页面和交互服务于用户任务、状态变化或风险控制 |
| jq-pm-review | 识别假设冒充事实、人为约束、无因果功能和过度方案 |
| jq-pm-briefing | 用目标、事实、约束、推导、验证解释重大决定 |
| jq-ui-spec | 从任务层级、信息、动作和反馈推导界面，减少惯性照搬 |

## 停止与边界

- 当目标、关键事实、硬约束、因果链、当前选择和最小验证均已明确时停止继续下钻。
- 当继续追问只会进入实现细节且不会改变方向、范围、风险或验收时，交给下游 Skill。
- 不用“从零思考”否定已经核验的用户证据、领域规律和组织责任。
- 不为展示思考过程堆叠抽象概念；最终结论必须能够落实到决定、需求、原型、评审意见或验证动作。
- 不把 AI 推导当成已确认业务事实；价值取舍、风险承担和组织责任仍由有权角色确认。
