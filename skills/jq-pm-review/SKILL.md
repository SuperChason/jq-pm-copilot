# 企业产品评审

给出“哪里有问题、为什么、改成什么、如何验证”的可执行评审意见。

## 能力说明

- **核心任务**：在正式评审、开发排期或上线验收前，主动暴露会影响产品价值和交付质量的问题。
- **最小输入**：待评审的 PRD、原型或方案；优先补充版本、里程碑、业务规则、术语、数据和技术约束。
- **主要产出**：评审结论、P0 至 P3 问题清单、具体修改建议、验证方式、待决策事项和复审条件。
- **评审原则**：先检查价值、范围和主流程，再检查规则、状态、权限、数据、异常、验收和界面细节。
- **能力边界**：不把个人审美偏好当成硬性缺陷，不输出无法定位或无法验证的空泛意见，不在仅请求评审时擅自改写原文件。
- **上游来源**：jq-pm-prd、jq-pm-prototype、jq-ui-spec、领域术语、研究结论和历史决策。
- **下游衔接**：重大未知项进入 jq-pm-grilling；问题回到对应 Skill 修正；目标和范围根本不清时退回 jq-pm-discovery；评审结论进入 jq-pm-briefing。

## 评审流程

1. 确认评审对象、版本、目标受众、里程碑和本次结论类型。
2. 按 ../jq-pm-grilling/references/product-context-protocol.md 读取相关 PRD、原型、术语表、调研、决定和假设，标出材料冲突。
3. 先检查价值与首期范围，再检查用户流程和业务闭环。
4. 逐项检查角色、规则、状态、权限、数据、异常、历史兼容和验收标准。
5. 有原型时走查关键任务，调用 jq-ui-spec 检查视觉和交互一致性。
6. 对会改变评审结论的重大未知项，读取 ../jq-pm-grilling/SKILL.md 和对应协议，进入评审拷问模式；其他材料缺口直接列入评审问题。
7. 区分已确认缺陷、材料缺失、评审假设和待决策事项，避免把推测写成确定问题。
8. 按严重度和根因合并问题，只保留能改变决策或提高交付质量的意见，并附下一阶段交接。

## 严重度

- P0 阻断：方向错误、合规或安全高风险、主流程无法完成，当前版本不能通过。
- P1 重要：核心规则、状态、权限、数据或验收缺失，开发测试容易产生不同理解。
- P2 改进：明显影响效率、一致性、可维护性或演示质量。
- P3 建议：低风险体验优化，不影响当前里程碑。

P0 与 P1 的分界以“当前主流程能否安全、确定地完成”为准。主流程无法完成、结果无法确定或存在高风险时判为 P0；补齐规则后可以按既定方向实施时通常判为 P1。存在争议时说明判级依据。

## 单条意见格式

- 位置：章节、页面或流程节点
- 问题：可验证的具体缺陷
- 影响：对用户、业务、研发、测试或上线的影响
- 建议修改：明确写出改什么、改成什么
- 验证方式：评审或验收时如何判断已解决

## 评审结论

输出通过、有条件通过或暂不通过，并列出必须在本里程碑解决的问题、可后置问题、待决策问题及建议责任人。不要把个人审美偏好包装成硬性缺陷。

默认按“结论—问题统计—阻断项—重要问题—待决策项—可后置项—复审条件”组织评审报告。责任归属按问题的最终确认权建议给产品、业务、设计、研发或测试负责人。

交付前检查：每条问题都包含准确位置、可验证缺陷、实际影响、具体修改方式和关闭条件；严重度与里程碑影响一致；总体结论能够直接支持“是否进入下一阶段”的决定。
