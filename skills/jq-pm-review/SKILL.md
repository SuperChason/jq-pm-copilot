# 企业产品评审

给出“哪里有问题、为什么、改成什么、如何验证”的可执行评审意见。

## 评审流程

1. 确认评审对象、版本、目标受众、里程碑和本次结论类型。
2. 读取相关 PRD、原型、术语表、调研和历史决定，标出材料冲突。
3. 先检查价值与首期范围，再检查用户流程和业务闭环。
4. 逐项检查角色、规则、状态、权限、数据、异常、历史兼容和验收标准。
5. 有原型时走查关键任务，调用 jq-ui-spec 检查视觉和交互一致性。
6. 对重大不确定性进行追问；需要持续拷问时读取 references/grilling-method.md。
7. 按严重度整理问题，只保留能改变决策或提高交付质量的意见。

## 严重度

- P0 阻断：方向错误、合规或安全高风险、主流程无法完成，当前版本不能通过。
- P1 重要：核心规则、状态、权限、数据或验收缺失，开发测试容易产生不同理解。
- P2 改进：明显影响效率、一致性、可维护性或演示质量。
- P3 建议：低风险体验优化，不影响当前里程碑。

## 单条意见格式

- 位置：章节、页面或流程节点
- 问题：可验证的具体缺陷
- 影响：对用户、业务、研发、测试或上线的影响
- 建议修改：明确写出改什么、改成什么
- 验证方式：评审或验收时如何判断已解决

## 评审结论

输出通过、有条件通过或暂不通过，并列出必须在本里程碑解决的问题、可后置问题、待决策问题及建议责任人。不要把个人审美偏好包装成硬性缺陷。
