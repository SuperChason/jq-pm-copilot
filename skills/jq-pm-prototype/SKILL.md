# 企业产品原型

原型用于验证产品决策和完整流程。默认产出可在浏览器运行的高保真 PC 端后台原型。

## 开始条件

1. 读取 PRD、发现简报、流程图、现有页面和项目技术约束。
2. 明确本次原型要验证的问题、目标用户、关键任务和评审范围。
3. 若关键业务规则缺失，先列出阻断项；能用可逆假设推进时显式标注。
4. 加载 jq-ui-spec 的设计令牌、组件、页面模式、交互状态和无障碍要求。

## 工作流程

1. 建立页面清单和页面间导航关系。
2. 为每个关键任务补齐默认、加载、空、错误、无权限、禁用、成功和部分完成状态。
3. 选择交付形态：已有工程内实现，或使用 assets/prototype-starter 创建单页 HTML 原型。
4. 先搭信息架构和主流程，再补视觉细节与辅助场景。
5. 使用真实业务语义的示例数据，避免 lorem ipsum 和无意义占位词。
6. 保证按钮、表单、筛选、分页、弹窗、抽屉和反馈可以实际点击演示。
7. 在浏览器中逐条走查关键路径，并检查 1440×900、1280×800 两种常见桌面视口。
8. 调用 jq-ui-spec/scripts/check-enterprise-ui.mjs 做静态检查，修复确定性问题。
9. 对照 PRD 记录一致项、原型新增假设、PRD 缺失项和设计取舍。

## 交付要求

- 入口清楚，评审人无需阅读代码即可打开。
- 关键任务形成完整闭环，失败和恢复路径可演示。
- 交互元素有 hover、focus、active、disabled 等必要状态。
- 原型中的新增业务规则必须标为假设，不能悄悄写回 PRD。
- 视觉遵循 jq-ui-spec；项目已有设计系统优先时，记录差异和采用理由。

## 输出

1. 可运行原型
2. 页面与状态清单
3. 关键演示路径
4. PRD—原型差异记录
5. 已知限制与待确认项

详细步骤读取 references/prototype-workflow.md；基础模板位于 assets/prototype-starter。
