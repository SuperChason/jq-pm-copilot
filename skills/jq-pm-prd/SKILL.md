# 企业 PRD

以项目事实为底座，用产品语言定义要实现的业务效果，让研发能够继续完成技术设计，让测试能够据此提取验证场景。

执行前读取 ../jq-pm-grilling/references/platform-capability-fallback.md。无法读取或写入项目文件时，以完整成稿和产品上下文交接块维持阶段衔接。

## 能力说明

- **核心任务**：把已确认的产品结论转化为业务、产品、研发和测试可以共同评审的生产级 PRD。
- **最小输入**：明确的目标用户、核心问题和版本范围；优先补充发现简报、研究结论、现状流程、业务规则、原型和已确认约束。
- **主要产出**：默认生成 Markdown 格式的正式 PRD，并附场景验证清单、影响范围、变更摘要、待确认问题和自检结论。只有用户明确要求 Word 时才生成 Word。
- **正文重心**：详细功能设计占据主要阅读注意力，直接说明功能用途、页面数据域、按钮与操作、业务判断、系统反馈、异常边界和验收结果。篇幅跟随需求复杂度，不设置字数或页数目标。
- **结构原则**：使用 references/prd-template.md 的 12 章作为通用骨架。一级、二级和必要的三级标题可以稳定；更细标题根据功能类型和实际风险选择、合并、改名或省略，不生成空章节和无内容表格。
- **产品边界**：PRD 不编写数据库字段、字段类型长度、接口地址与请求响应参数、数据表结构、技术组件、代码逻辑、部署拓扑或研发实现步骤。已确认的技术约束只有在影响用户体验、业务规则或验收结果时才进入正文。
- **上游来源**：jq-pm-discovery 的发现简报、jq-pm-grilling 的决定与产品上下文、jq-pm-research 的证据结论、已确认会议决策和存量项目材料。
- **下游衔接**：研发基于 PRD 继续完成开发设计；涉及页面时进入 jq-pm-prototype；需要企业界面基线时调用 jq-ui-spec；正式评审前进入 jq-pm-review。

## 工作流程

1. 按 ../jq-pm-grilling/references/product-context-protocol.md 读取项目索引、发现简报、决定、假设、术语、已有 PRD、原型、会议纪要和调研，标出主版本与日期。
2. 建立事实清单：已确认、存在冲突、待确认、已废弃。按协议处理材料权威顺序，正文只保留当前生效口径。
3. 判断需求是否达到编写门槛。目标用户、核心问题、本期范围或关键业务规则缺失时先调用 jq-pm-discovery。
4. 关键需求仅以历史方案、竞品做法或未经验证的因果假设为依据时，读取 ../jq-pm-grilling/references/first-principles-protocol.md 执行快速检查；无法说明目标、事实、硬约束和产品机制时，转回 jq-pm-grilling 深度推导。
5. 从 references/prd-template.md 选择本次有价值的章节。默认沿用 1—12 章顺序；确无内容且不影响评审、开发和测试的章节可以省略或合并，保持团队现有目录规范优先。
6. 先建立功能清单，再按一个清晰用户任务或一个完整业务目标拆分 `9.n`。避免按数据库对象、接口或研发模块拆功能。
7. 涉及页面时，从原型、截图或现有页面逐项盘点菜单、页签、工具栏、行内操作、批量操作、弹窗和抽屉中的全部用户可见操作，建立操作覆盖清单。界面材料缺失时只写已确认操作，并把会影响范围或验收的缺口列为阻断待确认。
8. 每个 `9.n` 先用一段简短文字说明功能用途，再补充界面图形、数据域说明和按钮/操作级功能说明。会改变数据、状态、流程或权限结果的操作单独展开；查询、重置、展开收起等低风险操作可以集中说明，但每个操作都必须被覆盖。
9. 数据域说明统一使用 references/prd-template.md 中的四列表格，写清用户可见名称、业务含义、必填规则以及录入或生成方式、来源、默认值、范围、格式、联动和约束；不扩展为研发字段字典。
10. 出现多角色交接、三个及以上连续步骤、条件分支、状态流转、回退或异常恢复时，优先绘制流程图或状态图；图后只补充关键条件和图中难以表达的规则，避免重复复述。
11. 将会改变范围、流程、规则、状态、权限或验收口径的未知项标为“阻断待确认”，写明影响、责任人和最晚确认点；不自行补造答案。
12. 检查术语一致性；必要时读取 references/domain-modeling.md。跨章节追溯优先使用功能编号和验收编号，小型需求不强制建立多套 ID。
13. 按 references/prd-review-checklist.md 自检，逐一核对功能清单、页面操作清单、按钮/操作级说明和验收场景，修复能够依据材料确定的问题。
14. 默认保存或返回完整 Markdown 主文档；输出变更摘要、待确认问题和下一阶段交接。用户明确要求 Word 时再生成并执行 Word 交付检查。

## 详细功能设计质量门

- **功能概述直接**：用一段话讲清谁在什么情况下使用、解决什么问题、完成后得到什么结果，不使用概述表格。
- **使用过程清楚**：入口、前置条件、用户动作、系统反馈和完成去向能够连成闭环。
- **页面表现明确**：说明页面展示的关键信息、用户可以执行的操作、操作条件和结果反馈；只描述产品可见要求。
- **数据域可实现**：数据域表格逐项写清业务含义、必填规则、录入或生成方式、来源、默认值、范围、格式、联动和约束，名称与页面一致。
- **操作覆盖到按钮**：菜单、页签、工具栏、行内操作、批量操作、弹窗和抽屉中的用户可见操作均已覆盖；不使用“支持增删改查”概括多个动作。
- **关键按钮形成闭环**：说明按钮用途、所在位置、操作对象、可见与可用条件、触发后的产品处理、成功反馈、业务结果、失败提示、恢复方式和后续去向。
- **差异操作分开说明**：单条与批量、保存与提交、删除与撤回等操作只要条件、处理或结果不同，就分别描述。
- **规则可以判断**：触发条件、判断逻辑、优先级、例外、边界值和结果明确；复杂条件按需使用决策表。
- **流程可以走通**：主流程、分支、角色交接、状态变化、取消回退和失败恢复完整；复杂流转使用流程图或状态图。
- **权限可以验证**：说明哪些角色可以查看或操作、可处理的数据范围，以及无权限时用户看到的结果。
- **异常可以恢复**：只覆盖真实相关的空数据、重复操作、并发变化、依赖失败、部分成功、历史数据等场景，并写清用户下一步。
- **验收可以复现**：测试能够根据前置条件、操作步骤和可观察结果编写用例，无需推测业务规则。
- **技术空间保留**：正文不替研发决定字段结构、接口协议、存储方式、组件选型和代码实现。

## 写作与排版要求

- 结论先行，使用业务人员能够直接理解的短句。背景介绍通常不超过 300 字。
- 能用一句话说清的内容不扩写成一段；能用一段说清的内容不拆成多张表。
- 功能概述必须使用简短文字。表格主要用于清单、对比、规则组合、状态差异、权限差异和验收场景。
- 页面需要展示、填写或选择的内容，用产品语言说明业务含义、来源、条件和反馈；不输出研发字段字典。
- 按钮和操作名称与界面保持一致。界面尚未定稿时使用稳定的业务动作名，并明确“界面文案待原型确认”。
- 使用“当……时，用户……；系统……；完成后……；失败时……”描述行为闭环，按实际复杂度拆句。
- 禁止使用“支持新增/修改/删除”“进行必要校验”“根据权限展示”“友好提示”“自动同步相关数据”“支持灵活配置”等无法直接判断结果的表述。
- 流程图服务于理解，覆盖起点、终点、角色、关键判断、主要分支和回退路径；正文只补充图中没有承载的内容。
- 明确本期不处理的边界。没有实际内容的细节章节直接省略，不用套话填充。
- Skill 模板、示例和通用说明保持中性，不写入使用方公司名称、内部品牌、客户名称和项目敏感信息。
- 默认生成 `.md` 文件；具备文件写入能力时优先保存到用户指定位置或当前项目的正式文档目录，缺少写入能力时返回可直接保存的完整 Markdown。
- 只有用户明确要求 Word 时，才调用文档处理能力生成 A4 竖版文档：设置正式独立封面、纯白简洁页眉、页码和克制的表格样式；交付前完成结构、内容和逐页渲染检查。

## 交付内容

1. 以详细功能设计为核心、章节按需选择的 Markdown 正式 PRD
2. 本次新增、修改、删除内容摘要
3. 影响范围：用户、业务流程、产品功能、权限、存量数据和关联系统
4. 阻断待确认问题：影响、责任人和最晚确认点
5. PRD 自检结论

涉及界面时，后续调用 jq-pm-prototype；需要正式压力测试时调用 jq-pm-review。

交付前检查：研发能够明确要实现的产品效果并据此开展开发设计，测试能够提取业务场景和预期结果，业务能够确认规则，产品能够判断范围。仍然依赖口头解释的内容继续补齐，研发内部实现细节留在开发设计中。
