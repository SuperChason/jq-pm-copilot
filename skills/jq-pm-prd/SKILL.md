# 企业 PRD

以项目事实为底座，产出能直接进入评审和研发协作的需求文档。

## 能力说明

- **核心任务**：把已确认的产品结论转化为业务、产品、研发和测试可以共同执行的正式需求。
- **最小输入**：明确的目标用户、核心问题和版本范围；优先补充发现简报、研究结论、流程、规则、数据和技术约束。
- **主要产出**：正式 PRD、验收标准、影响范围、变更摘要、待确认问题和自检结论。
- **完整性原则**：每项关键能力都描述角色、前置条件、主流程、状态、规则、权限、数据、异常和验收。
- **能力边界**：不替研发虚构技术实现，不静默合并冲突口径，不把未确认规则写成项目事实。
- **上游来源**：jq-pm-discovery 的发现简报、jq-pm-grilling 的决策账本与产品上下文、jq-pm-research 的证据结论、已确认会议决策和存量项目材料。
- **下游衔接**：涉及页面时进入 jq-pm-prototype；需要企业界面基线时调用 jq-ui-spec；正式评审前进入 jq-pm-review。

## 工作流程

1. 按 ../jq-pm-grilling/references/product-context-protocol.md 读取项目索引、发现简报、决定、假设、术语、已有 PRD、原型、会议纪要和调研，标出主版本与日期。
2. 建立事实清单：已确认、存在冲突、待确认、已废弃。按协议处理材料权威顺序，冲突未解决时不静默合并。
3. 判断需求是否达到编写门槛。目标用户、核心问题、首期范围或关键规则缺失时先调用 jq-pm-discovery。
4. 按 references/prd-template.md 编写或修订正文。
5. 对每项能力补齐角色、前置条件、主流程、状态变化、业务规则、权限、数据、异常和验收标准。
6. 检查术语一致性；必要时读取 references/domain-modeling.md。
7. 按 references/prd-review-checklist.md 自检，修复文档中可以依据材料确定的问题。
8. 输出主文档、变更摘要、待确认问题和下一阶段交接。

## 写作要求

- 结论先行，使用业务可理解的短句。
- 功能名称、字段名称、状态名称保持唯一口径。
- 验收标准写成可观察、可判定的条件。
- 复杂流程使用流程图、状态表或权限矩阵，保持最小够用。
- 无来源的重大业务规则只能进入待确认项。
- 明确首期不做事项，控制评审范围。

## 交付内容

1. 正式 PRD
2. 本次新增、修改、删除内容摘要
3. 影响范围：用户、流程、系统、接口、数据、权限和历史兼容
4. 待确认问题：影响、建议方案、责任人和截止点
5. PRD 自检结论

涉及界面时，后续调用 jq-pm-prototype；需要正式压力测试时调用 jq-pm-review。

交付前检查：研发能据此估算和实现，测试能据此设计用例，业务能据此确认规则，产品能据此判断范围。任何角色仍必须依赖口头解释才能执行时，继续补齐对应章节或列入阻断性待确认项。
